# NFTables 协议转发指南

## 🔍 NFTables 协议转发限制

### NFTables NAT 的技术约束

NFTables 在网络地址转换（NAT）层面存在协议版本限制：

```bash
# ✅ 支持的转发方式
IPv4 → IPv4 转发    # table ip nat
IPv6 → IPv6 转发    # table ip6 nat

# ❌ 不支持的转发方式
IPv4 → IPv6 转发    # 需要协议转换
IPv6 → IPv4 转发    # 需要协议转换
```

### 为什么不能跨协议转发？

1. **数据包结构差异**

   - IPv4: 32 位地址，特定头部格式
   - IPv6: 128 位地址，不同头部结构

2. **NAT 表限制**

   - `table ip nat` 只处理 IPv4 数据包
   - `table ip6 nat` 只处理 IPv6 数据包

3. **内核处理机制**
   - netfilter hooks 按协议版本分离处理

## 🛠 真正的跨协议转发解决方案

### 1. NAT64/DNS64 专用网关

**用途**: IPv6 网络访问 IPv4 服务的过渡技术

#### NAT64 工作原理

- **协议转换**: 在 IPv6 和 IPv4 之间进行数据包协议转换
- **地址映射**: 将 IPv6 地址映射到 IPv4 地址池
- **状态跟踪**: 维护连接状态，确保双向通信

#### DNS64 工作原理

- **DNS 合成**: 为没有 AAAA 记录的 IPv4 服务器合成 IPv6 地址
- **地址包装**: 将 IPv4 地址包装成特殊的 IPv6 地址格式 (如: 64:ff9b::192.0.2.100)
- **透明访问**: 让 IPv6 客户端认为目标是 IPv6 服务器

#### 完整工作流程

```bash
# 1. IPv6 客户端查询 IPv4 服务器
IPv6客户端 → DNS64查询: www.ipv4-only.com
DNS64服务器 → 合成AAAA记录: 64:ff9b::192.0.2.100

# 2. IPv6 客户端访问合成的 IPv6 地址
IPv6客户端 → 访问: [64:ff9b::192.0.2.100]:80
NAT64网关 → 提取IPv4地址: 192.0.2.100
NAT64网关 → 协议转换: IPv6包 → IPv4包

# 3. NAT64 网关代理访问 IPv4 服务器
NAT64网关 → IPv4服务器: 192.0.2.100:80
IPv4服务器 → 响应数据
NAT64网关 → 协议转换: IPv4包 → IPv6包
IPv6客户端 ← 透明接收: 完全不知道目标是IPv4
```

#### 典型部署场景

- **运营商网络**: IPv6-only 移动网络访问 IPv4 互联网
- **企业网络**: 新建 IPv6 网络需要访问遗留 IPv4 服务
- **数据中心**: IPv6 云环境访问 IPv4 应用服务器
- **过渡期管理**: 逐步从 IPv4 迁移到 IPv6 的中间方案

#### 实际配置示例 (Linux TAYGA)

```bash
# 安装 TAYGA NAT64 软件
apt install tayga

# 配置 /etc/tayga.conf
tun-device nat64
ipv4-addr 192.168.255.1
ipv6-addr 2001:db8:1::1
prefix 64:ff9b::/96
dynamic-pool 192.168.255.0/24

# 启动 NAT64 服务
systemctl enable tayga
systemctl start tayga

# 配置 DNS64 (BIND9)
# /etc/bind/named.conf.options
dns64 64:ff9b::/96 {
    clients { any; };
};
```

#### NAT64 地址转换规则

```bash
# IPv6 到 IPv4 地址映射示例
64:ff9b::192.0.2.1    ↔ 192.0.2.1
64:ff9b::8.8.8.8      ↔ 8.8.8.8
64:ff9b::172.16.1.100 ↔ 172.16.1.100

# 特殊前缀 64:ff9b::/96 是 RFC 6052 标准
# 后32位直接映射IPv4地址的二进制表示
```

## 🔄 跨协议转发方案对比

| 方案            | 工作层次 | 透明度   | 复杂度 | 性能 | 适用场景          |
| --------------- | -------- | -------- | ------ | ---- | ----------------- |
| **NAT64/DNS64** | 网络层   | 完全透明 | 高     | 优秀 | 运营商/大型网络   |
| **应用层代理**  | 应用层   | 部分透明 | 中等   | 良好 | 企业内部/特定服务 |
| **双栈隧道**    | 网络层   | 透明     | 高     | 良好 | 点对点连接        |
| **NFTables**    | 网络层   | 完全透明 | 低     | 优秀 | 同协议转发        |

## 💡 NAT64/DNS64 核心优势

### 为什么选择 NAT64/DNS64？

1. **完全透明**: IPv6 客户端无需任何修改，认为访问的就是 IPv6 服务器
2. **标准化**: RFC 6052、RFC 6144、RFC 6146 等标准支持
3. **运营商级**: 支持大规模部署，性能优秀
4. **自动发现**: DNS64 自动为 IPv4 服务合成 IPv6 地址

### 与 NFTables 的区别

```bash
# NAT64/DNS64: 真正的协议转换
IPv6客户端 → NAT64网关 → IPv4服务器
(不同协议包) → (协议转换) → (不同协议包)

# NFTables: 同协议地址转换
IPv4客户端 → NFTables → IPv4服务器
(相同协议包) → (地址替换) → (相同协议包)
```

## 🎯 总结

- **NFTables 适用于**: 同协议版本的高性能转发
- **NAT64/DNS64 适用于**: IPv6 网络访问 IPv4 服务的协议转换
- **应用层代理**: 灵活的跨协议转发，适合特定应用
- **最佳实践**: 根据实际需求选择合适的技术方案

我们的脚本通过自动协议匹配，确保在 NFTables 的能力范围内提供最佳的转发体验。对于真正需要跨协议转发的场景，建议使用专门的 NAT64/DNS64 或应用层代理解决方案。
